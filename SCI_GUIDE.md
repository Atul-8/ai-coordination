# ai-coordination 科普指南

> 面向所有开发者的架构思想与工具原理解读

## 这是什么？

**ai-coordination** 是一个 Claude Code 插件，它解决了一个 AI 辅助编程中的核心痛点：

**AI 会话终将结束，但你的项目还在继续。**

当你和 AI 一起写代码时，AI 掌握了丰富的上下文——你的需求、进度、遇到的坑、走过的弯路。但会话一结束，这些上下文就全部消失。下次对话，一切从零开始。

ai-coordination 就是给 AI 装上**持久化记忆**的插件。

---

## 核心问题：为什么需要它？

### 问题一：上下文断裂

AI 的上下文窗口是有限的。一次深度开发会话可能涉及几十个文件、数小时讨论，但：

- 会话超时 → 上下文丢失
- 切换话题 → 之前的工作状态遗忘
- 新会话启动 → 需要重新解释整个项目背景

**传统做法**：每次新会话手动告诉 AI "我上次做到哪了"。

**ai-coordination 做法**：AI 自动读取工作状态文件，1 秒恢复到上次中断点。

### 问题二：错误重复犯错

AI 在项目 A 中踩过的坑，在项目 B 中会再踩一次。因为每次会话的记忆是孤立的。

**传统做法**：希望 AI 自己记住教训（它做不到）。

**ai-coordination 做法**：错误知识库 + META 规则提炼，跨项目自动继承防线。

### 问题三：开发状态不可追溯

修改了什么？为什么这样改？需求何时变更？——全靠人脑记忆或散落在聊天记录中。

**传统做法**：翻聊天记录，问同事，对代码 diff。

**ai-coordination 做法**：操作履历自动记录，需求变更追踪，代码结构地图实时更新。

## 七层分治架构：一图看懂

```
                  coordination（对接层）
                  开发状态持久化、上下文容灾
                       ↓ 可读取所有层
    ┌─────────────────┼─────────────────┐
    ↓                 ↓                 ↓
presentation      interface          core
（展现层）        （接口层）          （核心层）
用户交互/视图     对外接口/协议适配    业务逻辑/核心算法
    ↓                 ↓                 ↓
    └─────────────────┼─────────────────┘
                      ↓ 均可依赖
                  shared（共享层）
                  常量、工具函数、通用类型、配置项
                  不依赖任何其他层，被所有层依赖
                      ↑ 被所有层依赖
                      ↓ 可测试所有层
                  testing（测试层）
                  各层接口验证、集成测试、回归守护
                  不被任何层依赖，可依赖所有被测层
                      ↑ 被所有层依赖
                  docs（文档层）
                  人与项目交互接口：使用者指南、需求梳理、数据对比
                  可读取所有层信息，禁止被任何代码层依赖
```

这套架构不是凭空发明的，它解决的是**软件工程的本质问题**——职责划分。

### 为什么是七层而不是三层？

传统三层架构（表现层/业务层/数据层）有两个盲区：**谁来管理开发过程本身？** 和 **谁来对接人？**

代码只管运行时不管开发时，也不管人怎么看。但开发过程和人员交互有大量重要信息：

- 我做到哪了？→ **工作状态**
- 我为什么这样做？→ **操作履历**
- 代码是怎么组织的？→ **结构地图**
- 需求怎么变的？→ **需求追踪**
- 踩过什么坑？→ **错误知识库**

这些信息不属于任何代码层，但它们对项目的**可持续性**至关重要。所以需要一个专门的 **coordination 层**来承载。

同时，项目需要与不同角色的人交互：

- 给**使用者**介绍项目怎么用 → **使用者指南**
- 给**产品经理**梳理业务需求 → **需求分析文档**
- 给**开发者**整理技术选型对比 → **数据对比报告**
- 给**新成员**解读项目架构 → **项目介绍**

这些"人与项目之间的数据交互"不属于任何运行时代码层，但对项目的**可理解性和可传播性**至关重要。所以需要一个专门的 **docs 层**来承载。

### 依赖规则为什么是严格的单向？

```
                  coordination（对接层）
                       ↓ 可读取所有层
    ┌─────────────────┼─────────────────┐
    ↓                 ↓                 ↓
presentation      interface          core
    ↓                 ↓                 ↓
    └─────────────────┼─────────────────┘
                      ↓ 均可依赖
                  shared（共享层）
              不依赖任何其他层，被所有层依赖
                      ↑ 被所有层依赖
                  testing（测试层）
              依赖所有被测层，不被任何层依赖
                      ↑ 被所有层依赖
                  docs（文档层）
              可读取所有层，禁止被任何代码层依赖
```

原因很简单：**防止循环依赖导致的混乱**。

- 上层可以调用下层——这符合"高层策略依赖低层实现"的设计原则
- 下层不能调用上层——否则就会形成循环，改一处动全身
- coordination 可以**读取**所有层信息用于记录状态，但**不能被运行时代码依赖**——它是元数据层，不是代码层
- docs 可以**读取**所有层信息用于生成文档，但**不能被运行时代码依赖**——它是人机交互层，不是代码层

> 类比：公司里的项目经理可以了解所有部门的情况来写报告，但各部门不需要依赖项目经理来干活。技术文档作者可以查阅所有系统来写手册，但系统运行不需要依赖文档。

## 核心结构：各自解决什么问题？

### 不增长文件（根目录）

#### 1. README.md — 目录说明「门牌」

新成员或新会话的第一站。描述 .ai/ 的结构、每个部分的用途、共享策略。一看就懂这里有什么、干什么用。

#### 2. WORKSTATE.md — 工作状态「进度条」

**解决的问题**：每次新会话，AI 不知道你做到哪了。

**工作方式**：
- 记录当前进行的任务和进度
- 维护未完成任务队列
- **最关键**：记录"上次中断点"——精确到文件和行号

当新会话启动时，AI 优先读取这个文件，直接从断点续接，就像你从未离开。

**类比**：游戏的"继续游戏"存档点。

#### 3. STRUCTURE.md — 代码结构「地图」

**解决的问题**：项目大了，连 AI 都容易迷路。

**工作方式**：
- 维护目录到七层架构的映射关系
- 记录每个关键模块的职责和依赖
- 架构变更时自动同步更新

**类比**：商场的楼层导览图——知道卖什么在几楼，不用盲目逛。

### 增长目录（独立文件夹）

#### 4. changelog/ — 操作履历「黑匣子」

**解决的问题**：为什么做了某个决策？改了什么？

**工作方式**：
- 每次关键操作实时追加到 LOG.md
- 标注操作类型：完成 / 决策 / 修复
- 关联涉及的文件
- 文件过大时自动归档到 ARCHIVE-YYYY-MM.md

**类比**：飞机的黑匣子——不事无巨细全记录，只记录关键节点。

#### 5. requirements/ — 需求追踪「账本」

**解决的问题**：需求改来改去，最后没人记得最初要什么。

**工作方式**：
- 每条需求独立文件（REQ-001.md、REQ-002.md...）
- 追踪每个需求的状态：进行中 / 完成 / 取消
- 记录需求变更的影响范围

**类比**：合同变更记录——每次修改都有据可查。

#### 6. errors/ — 错误知识库「免疫系统」

**解决的问题**：同样的错误在不同项目中重复出现。

errors 目录分为两层：

**raw/ — 原始记录**（项目内部，不建议共享）

每条错误独立文件，遵循**五步法**：

```
症状 → 根因 → 修复 → 规律提炼 → 二次提炼接口
```

这不是简单的"错了就改"，而是**自我进化**的过程：

| 步骤 | 做什么 | 为什么 |
|------|--------|--------|
| 症状 | 记录表面现象 | 知道发生了什么 |
| 根因 | 连问3个为什么 | 找到真正的原因（非表面原因） |
| 修复 | 记录修复方式 | 知道怎么解决 |
| 规律提炼 | 抽象出通用规则 | 理解这类错误的模式 |
| 二次提炼接口 | 生成 META 规则 | 在其他项目中预防同类错误 |

**distilled/ — 提炼成果**（跨项目共享的"错题本"）

只存放从 raw 中提炼出的 META 规则汇总。新项目启动时只需读取这一个文件，即可获得跨项目的错误防线。

**关键机制**：如果一个错误复现次数 >1，说明之前的规律提炼不够透彻，AI 会回溯补充。这就是**自我升级**。

**类比**：raw 是病历本（详细但不外传），distilled 是疫苗配方（只提炼精华，广为传播）。

## 为什么叫 META？词源与含义

### META 是什么意思？

META 来自希腊语 **μετά**（metá），意思是"关于...的..."、"超越"。在英语中，它作为前缀，表示**对事物本身的反思和抽象**。

这个词在科技领域随处可见，你可能已经接触过却没意识到：

| 常见词 | 含义 | 解释 |
|--------|------|------|
| **meta**data | 关于数据的数据 | 数据库里的字段定义，描述了数据本身的结构 |
| **meta**cognition | 关于认知的认知 | "思考自己如何思考"，学习中刻意反思 |
| **meta**programming | 关于编程的编程 | 用代码生成代码，比如宏、模板 |
| **meta**-analysis | 关于分析的分析 | 把多项研究的结果汇总再做分析 |

**规律**：加了 meta 前缀，就是从"参与其中"跳到"审视全局"。

### 在本插件中的含义

在 ai-coordination 中，META 规则就是**关于错误的错误规律**——不是记录"这个错误怎么修"，而是提炼"这类错误怎么防"：

```
一阶知识（具体）：这个 null 导致崩溃，我加了判断 → 修好了这一个
META 知识（抽象）：所有外部数据源返回值必须做 null/undefined 检查 → 防住这一类
```

从"修bug"到"防bug"，就是从一阶跳到了 meta 层。

### 为什么不叫"规则"或"经验"？

| 名称 | 为什么不够准确 |
|------|---------------|
| 规则 | 太泛——代码规范是规则，架构约束也是规则，无法区分 |
| 经验 | 太主观——"我觉得应该..."不是经验，经过提炼的可验证检查项才是 |
| 教训 | 太窄——教训暗示犯了错，但 META 规则也可以来自最佳实践的归纳 |
| **META 规则** | 精确——它是"关于规则的规则"，从具体错误中抽象出来的二阶知识 |

### 生活类比

你小时候被热水烫过，于是学会了"端杯子之前先试温度"。这个学习过程就是 META 提炼：

```
具体事件（一阶）：这杯水烫了我 → 我知道这杯水不能直接喝
抽象规律（META）：所有不明确温度的液体都要先试 → 我以后再也不会被烫
```

一次被烫 → 永远免疫。这就是 META 规则的价值：**让一次教训变成终身防线**。

## META 规则：错误免疫的跨项目传播

### 什么是 META 规则？

META 规则是从具体错误中提炼出的**可跨项目复用的预防检查项**。

举个例子：

| 具体错误 | META 规则 |
|----------|-----------|
| 项目 A 中因为忘记检查 null 导致崩溃 | META-NULL: 所有外部数据源返回值必须进行 null/undefined 检查 |
| 项目 B 中因为异步操作未做错误处理导致数据丢失 | META-ASYNC: 所有 async 函数必须包含 try-catch 或等效错误处理 |

### 怎么传播的？

```
项目 A 遇到错误 → 五步法分析 → 提炼 META-XXX 规则
                                    ↓
项目 B 启动 → 读取 distilled/meta-rules.md → 写入预置防线 → 不再犯同类错误
```

这就像人体的免疫系统——感染过一次病毒后产生抗体，下次遇到同类病毒自动防御。

### 为什么传统方法做不到？

传统代码审查是人治的：
- 人会忘记 → 规则不会
- 人有盲区 → 系统性检查覆盖全
- 规则不跨项目 → META 规则天生跨项目

## 云同步：多机协作的基础

### 为什么 `.ai/` 是独立的 Git 仓库？

`.ai/` 目录在项目内部，但拥有自己独立的 Git 仓库。这是刻意的设计：

1. **与项目代码解耦**：项目代码的分支/提交不影响元数据
2. **独立版本管理**：状态变更有自己的历史
3. **灵活的访问控制**：可以只共享代码，不共享状态（或反之）
4. **轻量同步**：只推送 `.ai/` 的变更，不触碰主仓库

### 工作方式

```
/ai-sync https://github.com/yourname/project-ai-state.git
```

执行后，`.ai/` 目录的变更会被提交并推送到指定远程仓库。你在办公室的电脑上同步的状态，回家后另一台电脑拉取即可续接。

## 实际使用场景

### 场景一：长周期项目

你在开发一个大型应用，每天都有新需求、新问题。用 ai-coordination：

1. `/ai-init` — 初始化对接层
2. 开发过程中，AI 自动记录工作状态和操作履历
3. 遇到 bug → `/ai-error` — 不仅修复，还提炼预防规则
4. 需求变更 → 自动记录影响范围
5. 每天下班 → `/ai-sync` — 同步到云端
6. 第二天 → AI 自动读取工作状态，接着昨天的进度继续

### 场景二：多项目并行

你同时维护 3 个项目：

1. 项目 A 中提炼出 META-ASYNC 规则
2. 项目 B 启动时，AI 读取项目 A 的 META 规则
3. 项目 B 中写异步代码时，AI 自动按 META-ASYNC 检查
4. 结果：项目 B 没有犯项目 A 的同类错误

### 场景三：团队协作

团队成员共享 `.ai/` 的远程仓库：

1. 成员甲记录了某个复杂 bug 的五步法分析
2. 成员乙在自己电脑上 `/ai:sync` 拉取最新状态
3. 成员乙直接看到 `distilled/meta-rules.md` 中的防线规则
4. 大幅减少重复踩坑时间

## 设计哲学

### 1. 元数据不是代码

coordination 层的信息是**关于代码的描述**，不是代码本身。这意味着：

- 它不参与编译和运行
- 它不会被运行时代码依赖
- 它独立演进，不影响应用行为

### 2. 声明式优于命令式

插件的逻辑用 Markdown 定义（技能、命令、模板），而非代码。好处：

- 人可读、可审查、可手动修改
- 不依赖特定编程语言
- AI 天然理解 Markdown 语义

### 3. 自我进化

错误知识库不是静态的——它会在复现时自我升级，在跨项目时传播免疫。这是传统文档做不到的。

### 4. 最小侵入

- 不修改你的代码
- 不影响你的构建流程
- 只在 `.ai/` 目录下工作
- 随时可卸载（`/ai-uninstall`）

## 常见疑问

### Q: 这和 Git 本身有什么区别？

Git 管理的是**代码版本**，ai-coordination 管理的是**开发过程信息**。两者互补：

| 维度 | Git | ai-coordination |
|------|-----|-----------------|
| 记录什么 | 代码变更 | 工作状态、决策理由、错误知识 |
| 粒度 | 文件级 | 任务级、需求级、知识级 |
| 谁在读 | 人 | AI（优先）和人 |
| 目的 | 版本追溯 | 上下文恢复 + 知识进化 |

### Q: 会拖慢 AI 响应速度吗？

不会。文件读取是毫秒级操作，且只在会话开始时读取一次。

### Q: 数据安全吗？

- `.ai/` 目录的数据只存在于你的项目和云端仓库
- 你完全控制同步到哪个远程仓库
- 不涉及任何第三方数据收集
- 推荐将 `.ai/` 加入 `.gitignore`（初始化时自动处理）

### Q: 可以只使用部分功能吗？

可以。各部分各自独立，你可以：
- 只用 WORKSTATE.md 做上下文恢复
- 只用 errors/ 做错误管理
- 只共享 distilled/ 不共享 raw/
- 不使用云同步功能

## 总结

ai-coordination 的核心价值可以用一句话概括：

**让 AI 的经验不再随会话消失，让一个人的教训成为所有人的防线。**

它通过七层架构中的 coordination 层和 docs 层，实现了：

1. **上下文容灾** — 会话中断，工作不断
2. **状态持久化** — 开发过程全程可追溯
3. **错误自我提炼** — 从错误中进化，越犯越少
4. **跨项目免疫** — 一处踩坑，处处免疫
5. **最小侵入** — 不改代码，不加依赖，随时卸载
6. **人机对接** — docs 层打通项目与人的信息通道，让项目对各类角色可理解、可传播
