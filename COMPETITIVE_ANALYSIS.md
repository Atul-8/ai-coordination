## 3. 各竞品深度分析

### 3.1 ECC / Everything Claude Code（216,507★）

**核心思路**：Agent 全栈性能操作系统。不是单一功能的工具，而是一个覆盖 Agent 生命周期全方位的"操作系统"——268 个技能、66 个代理、84 个命令、Hook 自动化、Instinct 持续学习、AgentShield 安全扫描、跨 7 个平台。Anthropic 黑客松获胜作品，230+ 贡献者，10 个月高频迭代。

**核心特性**：

- **Instinct 持续学习**：自动从会话中提取模式，生成可复用的"本能"技能（含置信度评分、导入/导出、进化）
- **Hook 全生命周期覆盖**：SessionStart/Stop 自动记录摘要，`ECC_HOOK_PROFILE` 环境变量控制严格度
- **Eval 验证循环**：Checkpoint vs Continuous Eval、Grader Types、pass@k 指标
- **跨 7 平台**：Claude Code、Codex、Cursor、OpenCode、Gemini、Zed、GitHub Copilot
- **Rust 控制面**（ECC 2.0 alpha）：桌面仪表盘 + Session 管理 + 守护进程
- **商业化**：ECC Pro $19/seat/mo，GitHub Marketplace 应用

**优势**：

- **功能最全** — 从行为规范到安全扫描到多智能体编排，应有尽有
- **生态最大** — 21 万星，230+ 贡献者，12 种语言生态
- **持续学习体系** — Instinct 机制是唯一接近 ai-coordination 错误进化的竞品
- **跨平台覆盖最广** — 7 个主流 AI 编程工具
- **企业级** — 安全扫描（AgentShield）、PM2 多服务编排、付费版
- **迭代极快** — 周/月级发版，v1→v2 十个月大版本

**劣势**：

- **巨型依赖** — npm 包 + SQLite 状态存储 + Rust 控制面 + Hook 脚本 + 本地服务
- **复杂度爆炸** — 268 个技能、66 个代理对大多数用户是过度的
- **Instinct 学习 ≠ META 错误免疫** — Instinct 从会话中提取模式，但没有五步法，没有根因分析，没有跨项目防线传播
- **卸载困难** — 多层安装（插件 + Hook + 状态 + 控制面）需专用卸载流程
- **人可审查性低** — 脚本/数据库/Rust 控制面对用户不透明
- **商业化** — 免费版功能裁剪，私有仓库需 $19/seat

**与 ai-coordination 的关系**：最全面的竞争对手，但定位完全不同。ECC 是"万能军刀"，ai-coordination 是"专业手术刀"。ECC 的 Instinct 学习最接近 ai-coordination 的错误进化，但 Instinct 做的是"模式提取"（从会话中发现可复用模式），而非"错误免疫"（从错误中提炼跨项目防线）。一个关注"AI 习惯"，一个关注"AI 免疫"。

### 3.2 andrej-karpathy-skills（176,683★）

**核心思路**：一个 CLAUDE.md 文件，通过四条原则约束 AI 行为——先想后写、简单优先、手术式修改、目标驱动。来源于 Karpathy 对 LLM 编程弱点的观察。

**解决的问题**：

- AI 默默认错假设就往下跑
- AI 过度复杂化代码和抽象
- AI 改了不该改的代码
- AI 缺乏可验证的成功标准

**优势**：

- **极简主义巅峰** — 一个文件、四条原则、零依赖
- Karpathy 背书带来的巨大社区影响力
- 原则普适：与任何工作流都不冲突
- 同时支持 Cursor（.cursor/rules/）

**劣势**：

- **纯行为规范，无状态持久化** — 会话结束就归零，不记得上次做到哪了
- **无错误管理体系** — 只有"别犯错"的劝诫，没有"犯了怎么进化"的机制
- **无跨项目能力** — 每个项目各自贴一份，不共享
- **纯靠 AI 自觉遵守** — 没有强制门禁或验证机制

**与 ai-coordination 的关系**：互补而非竞争。karpathy-skills 是"驾驶守则"，ai-coordination 是"行车记录仪 + 保险系统"。两者可叠加使用。

### 3.3 claude-mem（82,698★）

**核心思路**：跨会话持久记忆压缩系统。自动捕获 AI 的工具使用观察，用 AI 生成语义摘要，下次会话时自动注入相关上下文。核心是"记住 + 压缩 + 搜索"。

**核心特性**：

- Hook 自动捕获操作 → AI 语义压缩 → 未来会话自动注入
- 渐进展开（Progressive Disclosure）：分层读取，显示 Token 成本
- 语义搜索：mem-search 技能自然语言查询历史
- Web UI 实时查看记忆流（localhost:37777）
- 支持 Claude Code / Gemini CLI / OpenCode / OpenClaw

**优势**：

- **自动化程度极高** — 装完即用，无需手动操作
- 多平台支持（Claude Code、Gemini CLI、OpenCode）
- 语义搜索是杀手级功能——用自然语言查历史
- 30+ 语言本地化
- npm 一键安装

**劣势**：

- **重量级依赖** — Node.js + 本地 Worker 服务 + 数据库
- **记忆 ≠ 知识** — 记住"上次做了什么"但不会"提炼为什么出错"
- 需要常驻后台服务（localhost:37777）
- 安装复杂度远高于纯 Markdown 方案
- 压缩后的语义摘要可读性下降

**与 ai-coordination 的关系**：最直接的竞争对手。claude-mem 做"记忆持久化"，ai-coordination 做"知识进化"。claude-mem 记住事实，ai-coordination 提炼规律。一个横向（广度记忆），一个纵向（深度进化）。

### 3.4 Planning-with-Files（23,426★）

**核心思路**：Manus 式持久化文件规划。灵感来自被 Meta 以 $20 亿收购的 Manus AI——"Markdown is my working memory on disk"。通过三个文件（task_plan.md、findings.md、progress.md）在磁盘上持久化任务规划、研究发现和进度追踪，AI 上下文丢失后可通过读取文件恢复。v3.0 增加了自主模式和完成门禁。

**核心特性**：
- **三文件模式**：task_plan（阶段规划 + 进度）、findings（研究笔记）、progress（会话日志 + 测试结果）
- **Hook 全生命周期**：SessionStart/PreToolUse/PostToolUse/Stop/PreCompact 自动注入和同步
- **完成门禁（Gated Mode）**：5 个条件全部满足才允许结束会话，防止未完成任务退出
- **Session Recovery**：/clear 后自动恢复上次会话上下文
- **17+ 平台**：Claude Code、Cursor、Codex、Gemini CLI、GitHub Copilot、Kiro、Hermes、OpenCode 等
- **SHA-256 签名认证**：防止 plan 文件被篡改
- **并行计划隔离**：多个并行会话各自独立的 plan 目录
- **96.7% benchmark 通过率**（Sonnet 4.6）

**优势**：
- **规划导向最成熟** — 三文件模式经过 Manus 实战验证，理念有 $2B 收购背书
- **自动化程度高** — Hook 全生命周期覆盖，完成门禁强制任务收尾
- **跨平台覆盖极广** — 17+ 平台，是目前跨平台最多的 Claude Code 技能
- **安全认证** — SHA-256 签名防篡改，安全审计通过
- **A/B 测试验证** — 3/3 盲测胜出，有数据支撑效果
- **迭代极快** — 40+ 版本，v2→v3 演进清晰

**劣势**：
- **只做规划，不做错误进化** — findings 记录发现但没有五步法提炼，没有 META 规则
- **无跨项目免疫** — 每个项目独立 plan，不会把经验传播到其他项目
- **聚焦任务规划，缺失开发过程管理** — 无需求追踪、无错误知识库、无架构地图
- **Hook 脚本复杂度高** — Shell 脚本 + 完成门禁逻辑 + 签名认证，卸载不干净
- **功能单一** — 只解决"规划 + 进度"，其他开发过程痛点（错误管理、需求变更）不覆盖

**与 ai-coordination 的关系**：最相似的"文件持久化"思路，但关注点完全不同。Planning-with-Files 关注"任务规划和进度追踪"，ai-coordination 关注"开发过程管理和错误免疫"。前者回答"做什么、做到哪了"，后者回答"为什么出错、怎么不再犯"。两者底层理念一致（文件系统 = 持久记忆），可组合使用。

## 4. ai-coordination 的独特壁垒

### 4.1 分治思想的双重落地 — 五步法 + 七层架构

分治（Divide and Conquer）是软件工程最核心的思想之一：把复杂问题拆成独立子问题分别解决，再合并结果。ai-coordination 将分治在两个维度上系统化落地，形成完整的方法论体系：

#### 五步法 — 分治在错误分析上的纵向递进

将"一个错误怎么才能永远不再犯"这个复杂问题拆成五个递进子问题：

```
复杂问题：怎么确保不再犯同类错误？
                ↓ 分治
┌─────────────────────────────────────────────────┐
│ 第1步 症状       → "表层是什么"                    │ ← 信息收集子问题
│ 第2步 根因       → "深层为什么"（连问3个为什么）      │ ← 因因分析子问题
│ 第3步 修复       → "当下怎么修"                     │ ← 解决子问题
│ 第4步 规律提炼    → "这类错误的通用模式"              │ ← 抽象子问题
│ 第5步 二次提炼接口 → "跨项目可复用的预防检查项"       │ ← 泛化子问题（合并）
└─────────────────────────────────────────────────┘
                ↓ 合并
         META 规则 = 从具体到普遍的知识跃迁
```

每一步有明确的输入输出，上一步的输出是下一步的输入。这不是简单的"记一下错误"，而是**结构化的知识提炼流水线**。

| 竞品 | 错误处理方式 | 本质 |
|------|------------|------|
| ECC Instinct | 自动提取会话模式 | 无结构的概率归纳 |
| Planning-with-Files | 3-Strike Protocol（试3次不行升级） | 试错法，无分析 |
| pro-workflow | SQLite 记录 + 自纠正循环 | 有记录无提炼 |
| aprende-skill | 对话审查自动提炼 | 单步提炼，无分层递进 |
| **ai-coordination** | **五步法：症状→根因→修复→规律→META** | **分治递进，每步有明确输入输出** |

#### 七层架构 — 分治在项目组织上的横向分层

将"一个项目的代码怎么组织才不会乱"这个复杂问题拆成七个职责单一的层：

```
复杂问题：项目代码怎么组织才能职责清晰、依赖可控？
                ↓ 分治
                  coordination → 开发状态持久化、上下文容灾
                       ↓ 可读取所有层
    ┌─────────────────┼─────────────────┐
    ↓                 ↓                 ↓
presentation      interface          core
用户交互/视图     对外接口/协议适配    业务逻辑/核心算法
    ↓                 ↓                 ↓
    └─────────────────┼─────────────────┘
                      ↓ 均可依赖
                  shared → 跨层通用基础
              不依赖任何其他层，被所有层依赖
                      ↑ 被所有层依赖
                      ↓ 可测试所有层
                  testing → 各层接口验证、集成回归守护
              不被任何层依赖，可依赖所有被测层
                      ↑ 被所有层依赖
                  docs → 人与项目交互接口
              可读取所有层，禁止被任何代码层依赖
```

传统三层架构（表现/业务/数据）有两个盲区：**谁来管理开发过程本身？** 和 **谁来对接人？** coordination 层填补了第一个空白——它不参与编译运行，只负责开发过程的元数据管理。docs 层填补了第二个空白——它负责项目与各类人类角色（使用者、产品经理、开发者、新成员）之间的数据交互，包括使用指南、需求梳理、竞品对比、项目介绍等。shared 是**地基层**——零依赖、被所有层依赖，承载常量、工具函数、通用类型和配置项。testing 是**守门员层**——可测试所有被测层但不被任何业务层依赖，承载企业级自动化验证和回归守护。

| 竞品 | 项目组织方式 | 问题 |
|------|------------|------|
| ECC | 无架构约束，268 个 skill 按功能分类 | 分类 ≠ 分层，无依赖方向控制 |
| Planning-with-Files | 三文件模式（plan/findings/progress） | 只管任务层面，不管代码架构层面 |
| karpathy-skills | 无 | 纯行为规范，不涉及代码组织 |
| claude-mem | 无 | 纯记忆管理，不涉及代码组织 |
| **ai-coordination** | **七层分治 + 严格单向依赖** | **同时管开发过程、代码架构和人员对接** |

#### 两个分治体系的协同

```
七层架构（横向分层）          五步法（纵向递进）
┌──────────────────┐
│  coordination    │ ← 错误记录和 META 规则存放在这里
│  (.ai/ 目录)     │ ← 根因分析和提炼由这里驱动
│                  │ ← 架构变更同步更新 STRUCTURE.md
├──────────────────┤
│  presentation    │
├──────────────────┤
│  docs            │ ← 使用指南、需求梳理、竞品对比存放在这里
│                  │ ← 项目介绍和架构解读由这里输出
├──────────────────┤     具体错误
│  interface       │         ↓
├──────────────────┤     第1步 症状
│  core            │         ↓
├──────────────────┤     第2步 根因
│  shared          │         ↓
├──────────────────┤     第3步 修复
│  testing         │         ↓
└──────────────────┘     第4步 规律
                            ↓
                         第5步 META → 回到 coordination 层存储
```

七层架构提供了**错误知识的归属层**（coordination）和**人与项目的交互层**（docs），五步法提供了**错误知识的提炼路径**。横向分层确保架构不乱，纵向递进确保知识不丢。两者共同构成了 ai-coordination 的方法论基础。

## 7. 总结

**ai-coordination 是目前唯一实现"错误自我进化 + 跨项目免疫"的 Claude Code 插件。**

在星数最高的头部项目中：

- **ECC / Everything Claude Code**（216K★）做全栈 Agent 操作系统，最接近的是 Instinct 学习机制，但 Instinct 做的是"模式提取"而非"错误免疫"
- **andrej-karpathy-skills**（176K★）做行为规范，不做状态持久化和错误进化
- **claude-mem**（82K★）做记忆持久化，不做知识提炼和跨项目免疫
- **Planning-with-Files**（23K★）做任务规划和进度追踪，以文件持久化对抗上下文丢失，但 findings 只记录不提炼，无错误免疫能力
- **ai-coordination** 做错误进化 + 跨项目免疫，这是以上项目都没有覆盖的空白地带

值得注意的是，Planning-with-Files 与 ai-coordination 底层理念一致——"文件系统 = 持久记忆"，但关注点互补：前者回答"做什么、做到哪了"，后者回答"为什么出错、怎么不再犯"。两者可组合使用。

此外，ai-coordination 在零依赖纯 Markdown（ECC 需 npm + Rust + SQLite，Planning-with-Files 需 Shell Hook 脚本，而 ai-coordination 零门槛）、铁律级强制执行（比 karpathy-skills 更严格）、选择性团队共享（ECC 和 Planning-with-Files 都不具备）、企业级架构约束（七层分治 + 严格单向依赖，ECC 无架构约束）方面也有差异化优势。

### 企业级适配分析

七层架构天然覆盖企业级项目的核心关注点：

| 企业级关注点 | 七层架构如何覆盖 | 对应层 |
|------------|---------------|-------|
| **安全合规** | 鉴权/加密/限流/数据脱敏统一在接口层管控，下游层无法绕过 | interface |
| **审计追溯** | 操作日志、需求变更、错误知识库全程记录，coordination 层自动同步 | coordination |
| **环境隔离** | 环境变量管理、配置项集中在 shared 层，core 层零环境耦合 | shared |
| **监控告警** | 性能埋点、日志规范、异常体系作为基础设施，所有层统一接入 | shared |
| **团队协作** | 选择性共享（raw/ 不共享 + distilled/ 共享），.ai/ 独立 Git 仓库 | coordination |
| **代码质量** | 严格单向依赖防止循环耦合，改一层不影响其他层 | 全局 |
| **需求追踪** | 每条需求独立文件 + 状态追踪 + 影响范围记录 | coordination |
| **错误复盘** | 五步法强制分析 + 复现次数追踪 + 自我升级机制 | coordination |
| **需求沟通** | 产品需求梳理、PRD 输出、技术选型报告统一在文档层管控 | docs |
| **知识传承** | 项目介绍、架构解读、新人引导由文档层统一输出，降低人员流动成本 | docs |
| **自动化验证** | 各层有独立测试策略，代码变更自动触发对应层测试，防止回归 | testing |

ECC 的企业级优势在于安全扫描（AgentShield）和付费管理后台，但在**架构约束**层面反而是空白——268 个 skill 按"功能"分类而非按"层级"分层，没有依赖方向控制，这在大型团队中正是混乱的根源。ai-coordination 用七层分治从架构层面防止了这种混乱，testing 层更填补了企业全自动化开发中的验证闭环——代码变更后必须运行对应层测试，确保不引入回归。docs 层填补了企业与外部角色之间的信息通道——需求梳理、项目介绍、竞品对比不再散落在各处，而是有明确的归属层。

短板在于自动化程度（不如 ECC / claude-mem / Planning-with-Files 的 Hook 自动触发）、跨平台覆盖（不如 ECC 的 7 平台 / Planning-with-Files 的 17+ 平台）和功能广度（ECC 是全栈，ai-coordination 是单点穿透），这些是后续可迭代的方向。
